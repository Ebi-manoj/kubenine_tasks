
## Docker Runtime Mental Model 

When we run:

```bash
docker run busybox echo "hello world"
```

### What happens internally:

- Docker first checks if the busybox image is available locally  
- If not, it pulls the image from Docker Hub  

- Then Docker creates a container from this image  

- It runs the command:
  
  ```bash
  echo "hello world"
  ```

- BusyBox is a lightweight Linux image that already contains basic Unix utilities like `echo`, `ls`, etc.

- So the container starts with `echo` as the main process (PID 1)

- The command prints:
  
  ```
  hello world
  ```

- Once the command finishes execution, the main process ends  
- Since container life depends on the main process → the container exits  

---

### Logs understanding:

- The output (`hello world`) is written to STDOUT  
- Docker captures STDOUT and STDERR of the container  

- These logs are stored on the host machine (not inside the container)

- Default location:

```bash
/var/lib/docker/containers/<container_id>/<container_id>-json.log
```

- That’s why even after the container exits, we can still run:

```bash
docker logs <container_id>
```

and see the output  

---

### Key takeaway:

- Container runs only as long as its main process runs  
- If the process finishes → container stops  
- Logs are not inside container → they are managed by Docker on the host
