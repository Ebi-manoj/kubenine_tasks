
### 1. Why is reducing Docker image size important?

Reducing image size helps in faster build, push, and pull operations. It also reduces storage cost and improves deployment speed in CI/CD pipelines.
### 2. Why is a working image not automatically a good production image?

A working image is not automatically always good , it may have a large size or security vulnerabilities, the files and dependencies that doesn't needed for runtime .
A production good image should have optimized size and minimal files and dependencies need for the runtime.
### 3. What is .dockerignore, and why does it matter?

it is used to exclude unnecessary files from the build context. It helps prevent unwanted files like .git, cache, or env from entering the image.

### 4. How does .dockerignore help build performance?

It reduces the size of build context sent to Docker daemon. Smaller context means faster build and less resource usage.

### 5. Why does Dockerfile instruction order matter?

Docker uses layer caching, so instruction order affects rebuild time. If dependencies are installed before code copy, they won’t reinstall on every code change.

### 6. What is a multi-stage Docker build?

Multi-stage build uses multiple FROM statements to separate build and runtime stages. It helps keep only required files in final image. The main idea is to make the final image having only minimal dependencies that need for runtime , which helps to reduce the size .


### 7. How does a multi-stage build differ from a single-stage build?

Single-stage keeps everything in one image including build tools. Multi-stage separates build and runtime, so final image is cleaner and smaller.
### 8. What kinds of files should not end up in a production runtime image?

Files like source code not needed, test files, `.git`, cache files, and build tools should not be included. Only runtime dependencies and app files should exist.

### 9. What changed between the original and optimized image?

Optimized image had fewer unnecessary layers and smaller size. It also separated build dependencies from runtime files.
### 10. What did docker history help you understand?

It shows each layer created by Dockerfile instructions. It helps identify large layers and understand where optimization is needed. When we compare both images we can see that original image have more mb in the dependencies  layer but in the optimized image it only have few size , because we just copied only the dependencies in the final stage not kept entire things. 

### 11. Why should runtime behavior remain unchanged after optimization?

Optimization should only improve efficiency, not functionality. If behavior changes, it means something important was removed or broken.

### 12. How do optimized images help CI/CD and deployments?

Smaller images build faster and deploy quicker. It improves pipeline speed and reduces downtime during deployments.

### 13. Why does this task matter in real DevOps work?

In real systems, image size and efficiency directly affect cost, speed, and reliability. we must ensure images are production-ready, not just working.

### 14. What would happen if you optimized the image but forgot to verify the application?

The app might break in runtime without you noticing. This can lead to failed deployments and production issues.

It teaches how to build efficient, clean, and production-ready images. It also builds understanding of optimization, debugging, and validation practices.
