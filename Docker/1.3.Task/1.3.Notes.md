## RUN vs CMD in Docker

### RUN
- Executes during **image build (`docker build`)**
- Used for **installing dependencies / setup**
- Each RUN creates a **new image layer**
- Can use **multiple RUN instructions**

---

### CMD
- Executes during **container runtime (`docker run`)**
- Defines the **default command** for the container
- Does **not run during build**
- Only the **last CMD is used**
- Can be **overridden** at runtime

Example:
```bash
docker run image-name ls
```

---

## Binding Address in Containers (0.0.0.0 vs 127.0.0.1)

#### Problem
Applications inside containers may not be reachable from the host even when port mapping is configured.

---

### Default Behavior

#### Flask
- Default bind: `127.0.0.1`
- Scope: Loopback interface only (inside container)

#### Node.js (Express)
- Default bind: `0.0.0.0`
- Scope: All interfaces (externally reachable)

---

### Key Difference

- `127.0.0.1` → binds only to container's loopback interface
- `0.0.0.0` → binds to all network interfaces (loopback + eth0)

---

## Network Flow (Docker)

### With 0.0.0.0 (Correct)

Host (localhost:5000)
→ Docker Port Mapping (-p 5000:5000)
→ Container Network (eth0:5000)
→ Application (listening on 0.0.0.0)
→ Response

---

### With 127.0.0.1 (Failure)

Host (localhost:5000)
→ Docker Port Mapping
→ Container Network (eth0:5000)
→ ❌ Application NOT listening on eth0
→ Connection Refused

App is only listening on:
→ 127.0.0.1 (inside container)

---

## Why It Fails

- Docker forwards traffic to container IP (eth0)
- Application is bound only to loopback (127.0.0.1)
- Network interface mismatch → request dropped

---

## Solution

Explicitly bind to all interfaces:

```python
app.run(host="0.0.0.0", port=5000)
```

---

## Mental Model

- `127.0.0.1` → "internal access only"
- `0.0.0.0` → "externally accessible"
