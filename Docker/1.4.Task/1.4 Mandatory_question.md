### 6. What exactly was wrong in the original Dockerfile?

The Dockerfile is functionally correct for this specific application, but not correct from a production or best-practice perspective . I could able to build the image and run the container perfectly. But the issue i found was , it didn't followed a good practice that should follow in a production application such as , it is directly installing dependencies with pip install command and not specifying the version. This will cause a version drift and trouble in our application if it is installed other versions we are not expecting.
Also it is not using the caching layer mechanism of docker efficiently , we should install the dependencies first and then copy the app.py, otherwise any small change happen in the app.py will trigger the docker to install the dependencies again and rebuild the previous layer.
### 7. Why is installing dependencies from requirements.txt better than installing them ad hoc?

Because it will lock the version of dependencies correctly. There will not be any issues like version drift and our app work as expected in our expected dependencies versions. Also we can simply install all the dependencies with one command pip install requirements.txt , no need to type each installation step if our app have more dependencies. Also provide a proper documentation for our app about which all dependencies we need by just looking in the requirements.txt file. 
### 8. Why must required files be copied into the image explicitly?

Docker is not capable of magically do everything. It only works as per our instructions we wrote in the Dockerfile. So if our app need other files we need to explicitly tell it to copy the required files. 
### 9. What specific changes fixed the issue?

Actually, as i said the app was running successfully, but i fixed the issues like , original Dockerfile was not installing a locked version , because of no requirements.txt. Also it is not efficiently using the caching layer mechanism of docker . So what i did was 

FROM python:3.10-slim
WORKDIR /app

COPY requirements.txt /app/

RUN pip install --no-cache-dir -r requirements.txt

COPY app.py /app/

CMD ["python", "app.py"]

copy the requirements.txt first and install dependencies first . It will efficiently cache this layer and will not rebuild even if any small change happen in app.py. 

### 10. What is the difference between a build-time problem and a run-time problem?

Build-time problems occur during image creation (docker build), such as syntax errors, missing files, or dependency installation failures.
Run-time problems occur after the container starts (docker run), such as application crashes, port issues, missing environment variables, or incorrect configurations.
### 11. Why is docker logs important in container debugging?

Docker logs will shows the output of stdout or stderr of the container without directly going inside the container . The container logs will be stored in a host directory (/var/lib/docker/containers/container_id/json.log). So we can access it easily and debug what actually happened inside the container.
### 12. Why is docker inspect important in container debugging?

With this one single command , we will get all important information about the container such as metadata, container state , networking , log path , volumes , port mapping etc.
### 13. Why is docker exec useful during debugging?

Docker exec helps us to enter inside the container and check information , run any commands, see files ,config etc. This is useful in debugging to ensure all configuration and files are not tampered.
### 14. Why should you apply minimal fixes instead of rewriting everything blindly?

Minimal fixes is better , because it will give us better mental model of that issue and helps us to catch arising bugs easily comparing to rewriting everything blindly.
