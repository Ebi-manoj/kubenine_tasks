
### 1. What is the main process inside a Docker container?

The main process inside a container is the process having PID1. It is responsible for the lifecycle of the container . The container will be only alive as long as this process is running.

### 2. Why does a container stop when the main process exits?

As I said before , the container lifecycle depends on the main process, when the main process stops/finish its execution , the container will exit normally with Exit Code 0.

### 3. Why did the busybox container exit immediately?

Busybox container only have a short lived process (echo or any command) , it's finish it execution very quickly and as now there is no process inside the container , container also exit immediately.
### 4. Why did the nginx container keep running?

Nginx container have a long live running process inside , the nginx master process which is responsible for listening all the HTTP request , it should keep running in a foreground mode to do processing requests . So as long as the main process is running container also does.
### 5. What is the difference between a normal exit and a manual stop?

Normal exit is happen , if the main process of the container finish its execution normally and the container will exit normally with Exit Code 0.
Manual stop is like we are forcefully stopping the main process of the container by sending a SIGTERM signal, so the process will gracefully shutdown without any error . As now there is no process running inside the container , the container will also exit normally with Exit Code 0.
### 6. What does exit code 0 usually indicate?

Exit Code 0 usually indicate that the process inside the container terminated successfully without any errors and container also exited normally as long as it doesn't have any running process.
### 7. What is the purpose of docker logs in lifecycle debugging?

Dockers logs will give the output of the container stdout and stderr . It is very useful to understand the behavior of application , what happened actually inside that application.
### 8. What is the purpose of docker inspect in lifecycle debugging?

Docker inspect give us a complete metadata about the container , including container running state, OOMKilled status , Exit code and more . These are very useful to identify the root cause of the container crash or container stopped behaviors.
### 9. What is the purpose of docker top?

Docker top will list all the running process inside the container in a host perspective.
### 10. What is the difference between docker start and docker run?

Docker start is used for starting a stopped container only.
Docker run is used for creating a new container and start the created container.
### 11. What is the difference between docker stop and docker rm?

Docker stop will stop a running container by sending a SIGTERM signal to the main process for graceful shutdown.
Docker rm is used to remove the container which is stopped or exited.
### 12. What is the difference between docker start and docker restart?

Docker start is used to start a stopped container .
Docker restart is used to reboot a running container , which is equivalent to docker stop and docker start internally.
### 13. Why is it important to understand lifecycle state before debugging production containers?

Because debugging depends on knowing
- Whether the container crashed, stopped, or exited normally
- Lifecycle state helps identify the root cause quickly
### 14. How is a container lifecycle different from a virtual machine lifecycle?

Container lifecycle is tied to the main process of the container .Container only lives as its main process is alive or running.
Virtual machine lifecycle is tied to full operating system.
