## SIGTERM vs SIGKILL  
  
### Definition  
  
SIGTERM (Signal 15)  
- Graceful termination signal  
- Allows process to handle shutdown  
  
SIGKILL (Signal 9)  
- Forceful termination signal  
- Immediately stops process (handled by kernel)  
  
---  
  
#### Key Differences  
  
| Aspect | SIGTERM | SIGKILL |  
|-------|--------|--------|  
| Type | Graceful | Forceful |  
| Can be handled | Yes | No |  
| Cleanup possible | Yes | No |  
| Can be ignored | Yes | No |  
| Docker command | docker stop | docker kill |  
| Exit code | Usually 0 | 137 |  
  
---  
  
### Internal Flow  
  
SIGTERM (docker stop)  
Docker → SIGTERM → Process → Cleanup → Exit  
  
SIGKILL (docker kill)  
Docker → SIGKILL → Kernel → Process terminated immediately  
  
---  
  
### Docker Behavior  
  
docker stop  
- Sends SIGTERM  
- Waits ~10 seconds  
- If not stopped → sends SIGKILL  
  
docker kill  
- Directly sends SIGKILL  
  
---  
  
### When to Use SIGKILL  
  
- Application is stuck (infinite loop / deadlock)  
- Process not responding to SIGTERM  
- Graceful shutdown timeout exceeded  
- Emergency situations (high CPU, memory leak)  
- Zombie or hung process  
  
---  
  
### Risks of SIGKILL  
  
- Data corruption  
- Incomplete transactions  
- Logs not flushed  
- Open connections not closed


---
## docker start vs docker restart  
  
docker start  
- Used on STOPPED container  
- Moves: STOPPED → RUNNING  
- Starts the same process again  
- No stop involved  
  
docker restart  
- Used on RUNNING container  
- Moves: RUNNING → STOP → RUNNING  
- Sends SIGTERM (graceful stop) then starts again  
- Equivalent to: stop + start
