## Docker Networking — Core Notes

### 🔹 Why Custom Network is Needed

- Default bridge lacks reliable DNS (no name-based resolution)
- Requires using unstable container IPs (bad practice)
- Custom network provides:
    - automatic DNS (container name → IP)
    - logical grouping of services
    - better isolation
- Enables production-style service communication

---

### 🔹 Default Bridge (`docker0`)

- Created automatically by Docker
- Containers get IP (e.g., `172.17.x.x`)
- ❌ No proper service discovery (name resolution unreliable)
- Communication possible via IP only
- Not recommended for multi-service apps

---

### 🔹 Custom Bridge Network

docker network create app-net

- Creates isolated virtual LAN
- Containers can:
    - communicate via container name
    - avoid hardcoded IPs
- Uses internal DNS (`127.0.0.11`)
- Best practice for real-world apps

---

### 🔹 How Network Works Internally

- Docker creates:
    - Linux bridge (virtual switch)
    - veth pairs (container ↔ bridge)
- Each container gets:
    - its own IP
    - its own network namespace
- Communication:

container → veth → bridge → veth → container

---

### 🔹 When Container is Created

docker run ...

- Automatically attached to:
    - default bridge (if no network specified)
- Or custom network:

--network app-net

---

### 🔹 Communication Rules

- Same network → ✅ can communicate
- Different networks → ❌ cannot communicate
- One container can join multiple networks

---

### 🔹 Why `localhost` Fails

- Each container has its own localhost

localhost = itself (not other containers)
